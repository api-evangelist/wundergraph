<article><p>AI agents should not be given unrestricted access. They need scoped permissions based on what each MCP tool actually touches. The key point is that this already exists in your federated schema through @requiresScopes. You do not need a separate auth layer. The router should enforce these rules automatically for agents, the same way it does for any other client. If access is insufficient, the response should clearly indicate which scopes are required so the client can request them.</p><h2>Why MCP Servers Need Per-Tool OAuth Scope Enforcement</h2><p>When we first shipped the <a href="https://wundergraph.com/blog/graphql-persisted-operations-llms-mcp">Cosmo Router MCP server</a>, we were focused on getting the basics right — exposing GraphQL operations as MCP tools, handling sessions, making it work with Claude, Cursor, and Windsurf. Authorization was simple: forward the <code>Authorization</code> header to the GraphQL endpoint and let the existing router auth handle it.</p><p>Then we watched what happened in practice. Every agent that connected got access to every tool. A read-only analytics assistant and an admin agent shared the same permission surface. There was no way to say &quot;<code>get_employees</code> needs <code>employees:read</code> but <code>update_employee_mood</code> needs <code>employees:write</code>&quot; at the MCP layer.</p><p>That's how agent integrations quietly become over-privileged. The safe option is to mint a broad token and move on. The result is predictable: every tool is available, least privilege disappears, and your schema-level security model stops at the MCP boundary. The <a href="https://cosmo-docs.wundergraph.com/federation/directives/requiresscopes"><code>@requiresScopes</code></a> directives your schema already declares — the ones your router already enforces for human consumers — were invisible to the MCP server.</p><p>The usual answer would be to build another BFF, this time for agents. It starts small. A couple of tools, a little scope mapping, maybe some middleware. Six months later, it's another policy surface, another deployment, another thing that can drift from the schema, and another place security has to audit.</p><p>A new consumer type should not require a new authorization system. The graph already represents the business. It already has governance, observability, tracing, performance, and field-level security built in.</p><h2>How to Derive MCP Tool Scopes from Your GraphQL Schema</h2><p>The authorization requirements for every operation are already declared in the federated schema:</p><pre>type Query {
  topSecretFacts: [Fact!]! @requiresScopes(scopes: [[&quot;read:fact&quot;], [&quot;read:all&quot;]])
  employee(id: ID!): Employee @requiresScopes(scopes: [[&quot;read:employee&quot;, &quot;read:private&quot;], [&quot;read:all&quot;]])
}
</pre><p>The router already evaluates these rules for every GraphQL request from human consumers. The schema already knows the answer. The MCP server just wasn't reading it.</p><p>This matters because MCP tool authorization isn't an endpoint-level problem — it's a field-level problem. A single MCP tool maps to a GraphQL operation that can touch dozens of fields across multiple subgraphs, each with different authorization requirements. A tool that queries both <code>topSecretFacts</code> and <code>employee</code> needs a combined scope requirement derived from both fields. A tool that only queries <code>topSecretFacts</code> needs less. Per-endpoint authorization (one flat scope per tool) can't express this — you'd either over-privilege or under-privilege every tool. Per-field authorization via <code>@requiresScopes</code> means the scope requirements are computed from exactly which fields the operation touches.</p><h2>Introducing Per-Tool OAuth Scope Enforcement in Cosmo Router</h2><p>Cosmo Router now enforces per-tool OAuth scopes for MCP servers by deriving scope requirements directly from <code>@requiresScopes</code> in your federated schema. When scopes are missing, it doesn't just reject the request — it tells the client exactly what to ask for. An actionable <code>WWW-Authenticate</code> challenge, computed from the schema, so the agent can step up authorization on the same session without reconnecting and without a human in the loop.</p><p>In our <a href="https://wundergraph.com/blog/mcp-scope-step-up-authorization">previous post</a>, we described how our first implementation hit an infinite loop in the MCP TypeScript SDK, uncovered a tension between the MCP spec and RFC 6750, and contributed fixes to both. This post is the delivery of the capability we outlined there.</p><h3>What This Feature Does</h3><ul><li><strong>Derives per-tool scopes from the schema</strong> — the MCP server parses each operation at startup, identifies which fields it touches, and computes combined scope requirements from <code>@requiresScopes</code> directives. No additional config.</li><li><strong>Returns actionable scope challenges</strong> — when a token lacks required scopes, the router returns a <code>WWW-Authenticate</code> header with the exact scopes to request, picking the cheapest path through OR-of-AND requirements.</li><li><strong>Enables MCP step-up authorization</strong> — agents escalate permissions on the same session without reconnecting. Each HTTP POST carries its own token.</li><li><strong>Applies authorization at every layer of the MCP flow</strong> — connection, method, built-in tools, per-tool schema-derived scopes, and runtime checks for arbitrary GraphQL execution.</li><li><strong>Integrates with any OAuth provider</strong> — JWKS-based JWT validation works with Keycloak, Auth0, Okta, or any standards-compliant MCP authorization server.</li></ul><h2>How Scope Challenges Work</h2><p>Reading <code>@requiresScopes</code> and blocking requests was the easy part. The more interesting problem was what happens after the block.</p><p>The standard pattern for MCP OAuth authorization failures is a 403 with no actionable guidance. The client gets rejected and is left to figure out what went wrong. For a human developer, that's a solvable annoyance — read the error, check the docs, request a new token. For an AI agent running autonomously, it's a dead end. The agent can't proceed, can't self-correct, and can't escalate.</p><p>This is where least privilege usually falls apart. Rejecting the call is easy. The hard part is telling the client <strong>exactly which scopes to request</strong> so it can obtain a new token and retry, automatically, without a human in the loop.</p><p>For simple, flat scope requirements, this is trivial. But <a href="https://cosmo-docs.wundergraph.com/federation/directives/requiresscopes"><code>@requiresScopes</code></a> uses <strong>OR-of-AND</strong> semantics — and when an operation touches multiple protected fields, the requirements combine via Cartesian product. The challenge response needs to pick the right path through a potentially complex set of alternatives.</p><p>Consider a tool backed by an operation that touches two fields:</p><pre># Field A
topSecretFacts: [Fact!]! @requiresScopes(scopes: [[&quot;read:fact&quot;], [&quot;read:all&quot;]])

# Field B
employee(id: ID!): Employee @requiresScopes(scopes: [[&quot;read:employee&quot;, &quot;read:private&quot;], [&quot;read:all&quot;]])
</pre><p>The combined requirement for a tool that queries both fields is the Cartesian product:</p><table><thead><tr><th>Option</th><th>Required scopes</th></tr></thead><tbody><tr><td>1</td><td><code>read:fact</code> AND <code>read:employee</code> AND <code>read:private</code></td></tr><tr><td>2</td><td><code>read:fact</code> AND <code>read:all</code></td></tr><tr><td>3</td><td><code>read:all</code> AND <code>read:employee</code> AND <code>read:private</code></td></tr><tr><td>4</td><td><code>read:all</code></td></tr></tbody></table><p>Any one of these four AND-groups satisfies the requirement. But which one should the <code>WWW-Authenticate</code> challenge return?</p><h3>How the Router Computes the Cheapest Scope Upgrade Path</h3><p>The router doesn't guess. It evaluates which AND-group is cheapest for the client based on what the token already has.</p><p>If the client's token carries <code>read:employee</code> and <code>read:private</code>:</p><table><thead><tr><th>AND-group</th><th>Missing scopes</th><th>Count</th></tr></thead><tbody><tr><td><code>read:fact</code>, <code>read:employee</code>, <code>read:private</code></td><td><code>read:fact</code></td><td><strong>1</strong></td></tr><tr><td><code>read:fact</code>, <code>read:all</code></td><td><code>read:fact</code>, <code>read:all</code></td><td>2</td></tr><tr><td><code>read:all</code>, <code>read:employee</code>, <code>read:private</code></td><td><code>read:all</code></td><td>1</td></tr><tr><td><code>read:all</code></td><td><code>read:all</code></td><td>1</td></tr></tbody></table><p>Groups 1, 3, and 4 are tied at one missing scope. The router picks the first match and returns the <strong>complete AND-group</strong> in the challenge:</p><pre>HTTP/1.1 403 Forbidden
WWW-Authenticate: Bearer error=&quot;insufficient_scope&quot;,
                         scope=&quot;read:fact read:employee read:private&quot;,
                         resource_metadata=&quot;https://mcp.example.com/.well-known/oauth-protected-resource/mcp&quot;
</pre><p>The client now knows exactly what to request from the MCP authorization server. No guessing, no over-requesting, no under-requesting.</p><p>For pre-defined operations, the scope requirements are pre-computed at startup and cached per tool. The runtime cost is comparing the token's scopes against the pre-computed AND-groups — a handful of string comparisons.</p><p>This is the difference between authorization that blocks and authorization that enables. A router that supports <code>@requiresScopes</code> can reject a request when scopes are missing. A router that dynamically computes scope challenges can tell the client exactly how to fix the problem — turning a dead end into an MCP step-up authorization flow that resolves autonomously.</p><h2>Five Levels of MCP Scope Enforcement</h2><p>MCP gateway authorization isn't a single gate, and treating it like one is how over-privileging happens. The implementation enforces scopes at five additive levels. Each one is a gate, and a request must pass all applicable levels:</p><pre>HTTP Request
  → Level 1: Initialize (all requests — &quot;can this client connect at all?&quot;)
    → Level 2: Method (tools/list, tools/call — &quot;can this client discover or invoke tools?&quot;)
      → Level 3: Built-in tool (execute_graphql, get_schema — &quot;can this client use this built-in?&quot;)
        → Level 4: Per-tool (from @requiresScopes — &quot;does this client have scopes for this specific tool?&quot;)
          → Level 5: Runtime (execute_graphql only — &quot;does the arbitrary query touch protected fields?&quot;)
</pre><p>Levels 1–3 are configured by the platform team. They define org-wide baselines:</p><pre>mcp:
  oauth:
    enabled: true
    authorization_server_url: &quot;https://auth.example.com&quot;
    scopes:
      initialize: [&quot;mcp:connect&quot;]        # Baseline for all requests
      tools_list: [&quot;mcp:tools:read&quot;]      # Discovery
      tools_call: [&quot;mcp:tools:execute&quot;]   # Invocation
      execute_graphql: [&quot;mcp:graphql:execute&quot;]
      get_schema: [&quot;mcp:schema:read&quot;]
    jwks:
      - url: &quot;https://auth.example.com/.well-known/jwks.json&quot;
</pre><p>Levels 4 and 5 come from the schema itself. Schema owners declare <code>@requiresScopes</code> on fields, and the MCP server picks it up automatically. No coordination meeting. No separate config to keep in sync. The platform team controls the protocol-level gates. The schema owners control the data-level gates. Both are enforced by the same router, in the same request path, with the same observability.</p><h3>How Step-Up Authorization Works Without Reconnecting</h3><p>The Streamable HTTP transport — where each JSON-RPC request is a separate HTTP POST — makes token rotation natural. Each POST carries its own <code>Authorization</code> header. The session persists via the <code>Mcp-Session-Id</code> header.</p><pre>sequenceDiagram
    participant Agent as AI Agent
    participant AS as Authorization Server
    participant MCP as Cosmo Router&lt;br/&gt;(MCP Server)

    Note over Agent,MCP: Session established with minimal scopes
    Agent-&gt;&gt;AS: Authorize (scope: mcp:connect mcp:tools:read mcp:tools:execute)
    AS-&gt;&gt;Agent: token1

    Agent-&gt;&gt;MCP: tools/list [Bearer token1]
    MCP-&gt;&gt;Agent: 200 OK — tool list

    Note over Agent,MCP: Agent calls a read tool — insufficient scopes
    Agent-&gt;&gt;MCP: tools/call get_employees [Bearer token1]
    MCP-&gt;&gt;Agent: 403 Forbidden&lt;br/&gt;WWW-Authenticate: scope=&quot;employees:read&quot;

    Note over Agent,AS: Agent requests additional scope
    Agent-&gt;&gt;AS: Authorize (scope: + employees:read)
    AS-&gt;&gt;Agent: token2

    Agent-&gt;&gt;MCP: tools/call get_employees [Bearer token2, same Mcp-Session-Id]
    MCP-&gt;&gt;Agent: 200 OK — employee data

    Note over Agent,MCP: Agent calls a write tool — needs escalation again
    Agent-&gt;&gt;MCP: tools/call update_employee_mood [Bearer token2]
    MCP-&gt;&gt;Agent: 403 Forbidden&lt;br/&gt;WWW-Authenticate: scope=&quot;employees:write&quot;

    Agent-&gt;&gt;AS: Authorize (scope: + employees:write)
    AS-&gt;&gt;Agent: token3

    Agent-&gt;&gt;MCP: tools/call update_employee_mood [Bearer token3, same Mcp-Session-Id]
    MCP-&gt;&gt;Agent: 200 OK — mood updated
</pre><p>No reconnection. No session loss. The agent progressively acquires only the permissions it needs, when it needs them. The <code>Mcp-Session-Id</code> stays constant across all token changes — the session is decoupled from the credential.</p><h2>RFC 6750, MCP, and the Spec Gap We Found</h2><p>This is where the story gets interesting. When we <a href="https://wundergraph.com/blog/mcp-scope-step-up-authorization">first tested per-tool scope challenges</a> with the official MCP TypeScript SDK, we watched the browser open for re-authorization. Then open again. And again. Infinite loop.</p><p>The SDK's <code>streamableHttp.ts</code> overwrites the scope variable on each 403 challenge instead of merging it with existing scopes. Gaining <code>employees:write</code> loses <code>employees:read</code>. The next request fails, triggering re-authorization that loses <code>employees:write</code>. Back to square one.</p><p>We spent time tracing through the SDK code and the MCP spec to understand why. The root cause is a tension in the specification itself. <a href="https://datatracker.ietf.org/doc/html/rfc6750">RFC 6750</a> says the <code>scope</code> attribute in a 403 should describe the scopes needed for <strong>the requested resource</strong>. The MCP spec says servers should also include the client's <strong>previously granted scopes</strong>. These pull in different directions, and the SDK was implementing one interpretation while our server followed the other.</p><p>Our default behavior is what we believe is correct: the server returns only the scopes the operation needs, consistent with RFC 6750. For clients that replace rather than accumulate scopes, we added a single config override to bridge the gap:</p><pre>mcp:
  oauth:
    scope_challenge_include_token_scopes: true
</pre><p>When enabled, the server unions the operation's required scopes with the scopes already on the client's token. It's not elegant, but it exists for one reason: to interoperate with clients that don't accumulate scopes yet. We filed <a href="https://github.com/modelcontextprotocol/typescript-sdk/issues/1582">issue #1582</a> against the TypeScript SDK, and there's already a <a href="https://github.com/modelcontextprotocol/typescript-sdk/pull/1618">PR that adds client-side scope merging</a>. We also opened <a href="https://github.com/modelcontextprotocol/modelcontextprotocol/issues/2349">an issue against the spec itself</a>. When the ecosystem catches up, users simply remove the override.</p><h2>How RFC 9728 Protected Resource Metadata Helps MCP Clients</h2><p>There's a simpler path for well-behaved clients that don't want to deal with step-up at all. When OAuth is enabled, the router exposes a public metadata endpoint at <code>/.well-known/oauth-protected-resource/mcp</code> per <a href="https://datatracker.ietf.org/doc/rfc9728/">RFC 9728</a>. Clients discover the authorization server and all supported scopes upfront, without manual configuration:</p><pre>{
  &quot;resource&quot;: &quot;https://mcp.example.com&quot;,
  &quot;authorization_servers&quot;: [&quot;https://auth.example.com&quot;],
  &quot;bearer_methods_supported&quot;: [&quot;header&quot;],
  &quot;scopes_supported&quot;: [
    &quot;mcp:connect&quot;,
    &quot;mcp:tools:execute&quot;,
    &quot;mcp:tools:read&quot;,
    &quot;read:all&quot;,
    &quot;read:employee&quot;,
    &quot;read:fact&quot;,
    &quot;read:private&quot;
  ]
}
</pre><p>The <code>scopes_supported</code> field is computed automatically as the union of all configured static scopes and all scopes extracted from <code>@requiresScopes</code> directives. Well-behaved clients request all supported scopes upfront during the initial authorization — the authorization server decides which to grant. MCP step-up authorization challenges serve as a fallback when the client doesn't have everything upfront.</p><h2>Why This Is Better Than Building Another BFF</h2><p>We've seen what happens when teams build a separate BFF for agents. It starts small — a couple of tools, a little scope mapping, maybe some middleware. Then the problems arrive:</p><ul><li><strong>Auth policy drifts from schema policy.</strong> The BFF's scope config is a manual copy of what the schema already declares. They diverge silently. Nobody notices until an audit.</li><li><strong>Platform teams duplicate governance.</strong> A new authorization layer means new audit surface, new monitoring, and new incident response playbooks. Another config file is not a security model.</li><li><strong>Agents become a separate security domain.</strong> Different rules for agents than for human consumers, maintained by different teams, validated by different processes. If your schema already declares the rules, copying them into another layer is not architecture. It's drift waiting to happen.</li><li><strong>Every new consumer type creates another control surface.</strong> After agents, it's workflow engines, partner integrations, internal tools — each with its own BFF.</li></ul><p>The router already enforces <code>@requiresScopes</code> for GraphQL consumers. Adding MCP authorization to the same router means agents get the same security model — one policy, one enforcement point, one thing to monitor.</p><h2>Real-World MCP Authorization Scenarios</h2><h3>External Partner Gets Read-Only MCP Access</h3><p>A partner integration team connects their AI assistant to your API for customer support. Their OAuth client is configured with <code>mcp:connect</code>, <code>mcp:tools:read</code>, <code>mcp:tools:execute</code>, and <code>orders:read</code>. Your schema has <code>@requiresScopes(scopes: [[&quot;orders:read&quot;]])</code> on order query fields and <code>@requiresScopes(scopes: [[&quot;orders:write&quot;]])</code> on mutation fields. The partner's agent can call <code>get_order_status</code> freely. If it tries to call <code>cancel_order</code>, it gets a 403 with <code>scope=&quot;orders:write&quot;</code> — a scope the partner's OAuth client isn't authorized to obtain. The router enforces the boundary.</p><h3>Internal Agent Gets Progressive Write Access</h3><p>Your internal AI assistant needs broad read access but should only acquire write scopes when a user explicitly approves a destructive action. The agent starts with read scopes and reads data freely. When a user says &quot;delete that deployment,&quot; the agent calls <code>delete_deployment</code>, gets a 403 with <code>scope=&quot;deployments:delete&quot;</code>, surfaces this to the user, triggers a consent flow via the authorization server, obtains a scoped-up token, and retries — all on the same session.</p><h3>Platform Team Governs 50 Services Without Per-Service Config</h3><p>The platform team sets levels 1–3 once: <code>mcp:connect</code> to connect, <code>mcp:tools:read</code> to discover, <code>mcp:tools:execute</code> to invoke. Individual service teams declare <code>@requiresScopes</code> on their fields as they already do. When a new team onboards, the MCP server picks up their scope requirements automatically on the next config reload. No MCP-specific authorization config to maintain.</p><hr /><p>We started this work because we watched agents connect with broad tokens and call every tool indiscriminately. The authorization model we'd built for human consumers — field-level, scope-based, declared in the schema — wasn't reaching the MCP layer.</p><p>Now it does. Agents should not need god tokens just because the transport is new. Per-tool OAuth scopes for MCP close the loop — agents get least-privilege access, actionable scope challenges, and step-up authorization on the same session, all derived from the same <code>@requiresScopes</code> directives that govern every other consumer. No new BFF. No separate config. No drift.</p><h2>What's Next</h2><ul><li><strong>Search &amp; Execute</strong> — tool discovery by intent, reducing token consumption</li><li><strong>Code-mode</strong> — natural language to GraphQL execution on the MCP server</li><li><strong>Advanced Prompt-to-Query</strong> — bridging the gap where LLMs can't model your business domain from a schema alone</li></ul><h2>Get Started</h2><ul><li><a href="https://cosmo-docs.wundergraph.com/router/mcp">MCP Gateway Documentation</a> — Full setup guide and configuration reference</li><li><a href="https://cosmo-docs.wundergraph.com/router/mcp/oauth">OAuth 2.1 Authorization</a> — MCP scope enforcement, JWKS config, step-up authorization</li><li><a href="https://cosmo-docs.wundergraph.com/federation/directives/requiresscopes"><code>@requiresScopes</code> Directive</a> — Declaring per-tool authorization in your schema</li><li><a href="/blog/cosmo-mcp-automates-graphql-federation-development">Automate GraphQL Federation Development with MCP</a> — Use the Cosmo MCP server for schema management and development workflows</li><li><a href="https://github.com/wundergraph/cosmo">GitHub: Cosmo</a> — Open-source federated GraphQL platform</li></ul><hr /><hr /></article>
