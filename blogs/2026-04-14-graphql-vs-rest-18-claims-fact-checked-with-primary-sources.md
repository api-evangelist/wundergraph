A fair comparison requires evaluating both technologies under equivalent assumptions: default vs. default and optimized vs. optimized.</p><p>In this analysis, “default” means out‑of‑the‑box behavior, while “production” means commonly adopted patterns in mature or scaled deployments.</p><p>Where an original claim is right, I say so; where the evidence disagrees, I show why.</p><h2>The Origin Story: How Facebook Actually Built GraphQL</h2><h3>Claim 1: &quot;Facebook built GraphQL for hundreds of internal microservices&quot;</h3><p><strong>Verdict: Historically imprecise.</strong></p><p>The 2012 timeline, the iOS News Feed context, and the mobile bandwidth constraints are all real.</p><p>I could not find a primary source describing Facebook's 2012 backend as a system composed of &quot;hundreds of microservices,&quot; or stating that GraphQL was created to unify such a system. The <a href="https://graphql.org/learn/federation/">GraphQL Foundation's own Federation page</a> states:</p><blockquote><p>Meta (formerly Facebook), where GraphQL was created, has continued to use a monolithic GraphQL API since 2012.</p></blockquote><p>Facebook ran a massive PHP codebase, scaled through <a href="https://engineering.fb.com/2011/12/09/open-source/the-hiphop-virtual-machine/">HipHop</a> (a PHP-to-C++ transpiler) and later <a href="https://hhvm.com/">HHVM</a> (a just-in-time compiler for PHP). <a href="https://github.com/kmapb">Keith Adams</a>, Facebook's Chief Architect and HHVM team lead, describes how Facebook scaled a large, unified PHP application in <a href="https://softwareengineeringdaily.com/2019/07/15/facebook-php-with-keith-adams/">Software Engineering Daily interviews</a> and <a href="https://www.infoq.com/interviews/adams-php-facebook/">InfoQ presentations</a>.</p><p>The strategy emphasized scaling PHP through compilation rather than prioritizing early service decomposition. On top of the PHP application sat <a href="https://engineering.fb.com/2013/06/25/core-infra/tao-the-power-of-the-graph/">TAO</a>, a graph-structured data layer built on MySQL and memcached. TAO provided a unified way to access social graph data. The backend included multiple systems and data sources, but primary sources do not describe it as a microservice-oriented architecture.</p><p>Meta's <a href="https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/">engineering blog</a> describes &quot;existing databases and business logic,&quot; rather than framing the problem as a microservice-oriented API layer. The common argument is that GraphQL was built to manage hundreds of microservices. There is no primary source clearly supporting that framing of the original problem GraphQL was designed to solve, and its later use in microservice architectures does not change that original motivation.</p><p>In short, microservices later became a use case for GraphQL, not the original design driver; saying GraphQL was “built for hundreds of microservices” reverses that causal order.</p><h3>Claim 2: &quot;Lee Byron, Dan Schafer, and Nick Schrock built GraphQL&quot;</h3><p><strong>Verdict: Generally accurate, but slightly simplified.</strong></p><p>Primary sources and historical accounts consistently credit <a href="https://github.com/leebyron">Lee Byron</a>, <a href="https://github.com/dschafer">Dan Schafer</a>, and <a href="https://github.com/schrockn">Nick Schrock</a> as the key creators of GraphQL. Nick Schrock is widely described as writing the initial prototype, reportedly called &quot;SuperGraph.&quot; Together with Dan Schafer and Lee Byron, they developed GraphQL and brought it into production at Facebook around 2012.</p><p><a href="https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/">Nordic APIs interviewed Lee Byron</a> about the creation process, and <a href="https://www.datanami.com/2019/06/27/graphql-creators-provide-a-tutorial-on-teamwork/">Datanami</a> covers the contributions of all three.</p><p>However, this framing simplifies the broader effort. GraphQL was developed within a larger team at Facebook, and multiple engineers contributed to its design, implementation, and adoption. The claim is accurate at a high level, but should be understood as identifying the primary contributors rather than the full set of people involved.</p><h3>Claim 3: &quot;Facebook open-sourced GraphQL in 2015&quot;</h3><p><strong>Verdict: Accurate.</strong></p><p>Facebook publicly released GraphQL on <a href="https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language">September 14, 2015</a>, following its public introduction at the React Europe conference in Paris earlier that year. In 2018, GraphQL was <a href="https://techcrunch.com/2018/11/06/facebooks-graphql-gets-its-own-open-source-foundation/">moved to the GraphQL Foundation</a>, where it is now governed under the <a href="https://www.linuxfoundation.org/">Linux Foundation</a>.</p><h2>The GraphQL N+1 Problem: Does REST Have It Too?</h2><h3>Claim 4: &quot;Fetching 25 users with posts in GraphQL results in 26 database queries&quot;</h3><p><strong>Verdict: Partially true (implementation-dependent, not inherent to GraphQL).</strong></p><p>The arithmetic is correct for one specific scenario, and the official <a href="https://www.graphql-js.org/docs/n1-dataloader/">GraphQL.js documentation</a> confirms it:</p><blockquote><p>In GraphQL.js, each field resolver runs independently. There's no built-in coordination between resolvers, and no automatic batching.</p></blockquote><p>This behavior is not defined by the GraphQL specification, but by how resolver-based servers like GraphQL.js execute queries. When you query for 25 users and their posts, the server may first execute one query to fetch all 25 users. Then, for each user, it may execute a separate query to fetch that user's posts. That's 25 + 1 = 26 database queries, which is the classic &quot;N+1 problem&quot;: N queries for related data, plus 1 for the initial list.</p><p>For servers built on GraphQL.js without DataLoader, this is a common outcome in resolver-based implementations. In practice, many GraphQL deployments introduce batching layers, such as DataLoader, or use engines that compile queries into optimized database queries to avoid this pattern. But the picture is incomplete in two ways.</p><p>First, some GraphQL engines addressed this years ago. Tools like <a href="https://hasura.io/blog/architecture-of-a-high-performance-graphql-to-sql-server-58d9944b8a87">Hasura</a> and <a href="https://postgraphile.org/">PostGraphile</a> don’t rely on per-field resolvers for database-backed queries. They compile GraphQL queries directly into optimized SQL with JOINs. In these systems, the 25-users-with-posts query can often be executed as a single database query. In its own benchmarks, Hasura reports up to <a href="https://hasura.io/blog/compiling-graphql-for-optimal-performance-going-beyond-dataloader">higher throughput</a> than hand-written Node.js code using DataLoader. This is based on vendor benchmarks and should be treated as directional rather than conclusive.</p><p>Second, REST can exhibit a similar N+1 pattern at the network layer, as shown in the next claim.</p><h3>Claim 5: &quot;In REST, fetching 25 users with their posts is two calls&quot;</h3><p><strong>Verdict: Misleading.</strong></p><p>A common claim is that REST handles this in two calls:</p><p><code>GET /users</code> and <code>GET /users/:id/posts</code>.</p><p>But look at that second URL. The <code>:id</code> is a path parameter.</p><p>Without a dedicated batch endpoint, the REST client has to call <code>GET /users/1/posts</code>, <code>GET /users/2/posts</code>, ... <code>GET /users/25/posts</code>. Without a batch or embedding mechanism, this can result in 26 HTTP round-trips. In many systems, 26 HTTP round-trips (each requiring a full network request-response cycle) is going to be more expensive than 26 database queries behind a single HTTP call.</p><p>The <a href="https://restfulapi.net/rest-api-n-1-problem/">REST API Tutorial</a> documents this exact problem. Microsoft's Azure Architecture Center calls this the <a href="https://learn.microsoft.com/en-us/azure/architecture/antipatterns/chatty-io/">Chatty I/O antipattern</a> and explicitly names it &quot;the N+1 problem.&quot; N+1 is not a GraphQL-specific problem. It manifests at different layers.</p><p>To get it down to two calls, REST needs a purpose-built batch endpoint like <code>GET /users/posts?ids=1,2,...,25</code>. That endpoint either uses POST (typically uncacheable in practice) or GET with dynamic query parameters where every unique combination of user IDs produces a different URL.</p><p>And it's not just the combination that matters; it's also the order, unless it is normalized.</p><p><code>?ids=1,2,3</code> and <code>?ids=3,1,2</code> are different URLs to a CDN, which means different permutations of the same IDs can produce different cache keys. In practice, many REST architectures introduce aggregation layers or backend-for-frontend services to avoid this pattern. We'll come back to why this matters in the caching section.</p><p>The fair comparison is either naive-to-naive (26 HTTP calls vs. 26 DB queries) or optimized-to-optimized (batch endpoint vs. DataLoader or query compilation, both producing 1-2 queries). The original claim oversimplifies the design trade‑offs by mixing optimized REST with naive GraphQL.</p><h3>Claim 6: &quot;DataLoader is not built into GraphQL and is not part of the specification&quot;</h3><p><strong>Verdict: True (but often misunderstood in practice).</strong></p><p>The statement is technically correct at the specification level, but incomplete in how GraphQL is used in practice.</p><p>DataLoader was originally developed at Facebook, and its copyrights were later transferred to the GraphQL Foundation. It is now maintained as part of the broader GraphQL ecosystem. It batches and deduplicates data fetches within a single request, does not share results across requests, and is not a replacement for application-level caching.</p><p>When multiple resolvers request the same data (e.g., &quot;fetch user 1&quot;, &quot;fetch user 5&quot;, &quot;fetch user 12&quot;), DataLoader collects them into a single batch call and issues one combined fetch. Instead of 25 individual database queries, a single batched query can often fetch all 25 users at once.</p><p>DataLoader is generic and not specific to GraphQL, the same pattern can be used in REST APIs, microservices, or any system with N+1-style access patterns.</p><p>For example, <a href="https://netflix.github.io/dgs/data-loaders/">Netflix DGS</a> provides <code>@DgsDataLoader</code> annotations in Java,  <a href="https://gqlgen.com/reference/dataloaders/">gqlgen</a> documents DataLoader integration for Go, and <a href="https://graphql-ruby.org/">GraphQL-Ruby</a> has <code>GraphQL::Dataloader</code> built in. Query compilation engines (Hasura, PostGraphile) bypass the need for DataLoader by generating optimized SQL directly.</p><p>DataLoader reduces redundant fetches, but it introduces additional complexity in resolver design and requires careful scoping and key management.</p><p>REST does not define a standard batching mechanism at the protocol level. When batching is needed, it is handled through API design (such as bulk endpoints), infrastructure, or framework-specific solutions. Some specifications attempt to address this, such as <a href="https://www.odata.org/">OData’s</a> batch format or <a href="https://jsonapi.org/">JSON:API’s</a> compound documents, but adoption is inconsistent.</p><p>These differences reflect where batching is handled. GraphQL solutions often address it at the execution layer, while REST implementations handle it through API design or infrastructure. DataLoader is therefore just one of several batching patterns in the GraphQL ecosystem, not the only way to avoid N+1.</p><h2>GraphQL vs REST Performance: What Benchmarks Actually Show</h2><h3>Claim 7: &quot;REST delivers nearly half the latency and 70% more requests per second&quot;</h3><p><strong>Verdict: Context-dependent and often misapplied.</strong></p><p>These numbers come from a <a href="https://medium.com/jds-engineering/i-was-curious-which-is-faster-rest-or-graphql-so-i-tested-themto-say-hello-world-d6c28fec4f17">Medium benchmark</a> that tested a Hello World‑style endpoint and is representative of a broader class of “Hello World” REST vs GraphQL benchmarks, not necessarily the exact one cited by any particular critic:</p><p>REST at ~7.68ms latency vs. GraphQL at ~13.51ms, and ~11,972 RPS vs. ~7,085 RPS.</p><p>That kind of workload mostly measures protocol overhead on a flat response, not the multi-resource retrieval problem GraphQL was designed to address. It is also informal evidence from a single implementation, not a peer-reviewed study. For relational or multi-resource data retrieval, several studies report the opposite result under some workloads.</p><p>A peer-reviewed study by Seabra, Nazário, and Pinto, published at <a href="https://dl.acm.org/doi/10.1145/3357141.3357149">SBCARS '19</a> (proceedings in the ACM Digital Library), found that migrating from REST to GraphQL reduced latency in two of three tested applications, often by reducing the number of network round-trips. Under workloads above 3,000 requests, REST outperformed GraphQL.</p><p>A study by Jin, Cordingly, Zhao, and Lloyd at <a href="http://faculty.washington.edu/wlloyd/papers/graphql-camera-ready.pdf">UW Tacoma</a>, presented at WoSC10 (December 2024), evaluated a 10.9‑million‑row CMS dataset on AWS, and reported <strong>25–67%</strong> lower average latency for GraphQL on several data-intensive operations, while REST retained an advantage at the highest concurrency levels.</p><p>A <a href="https://doaj.org/article/d3438a3cda37412aa9a755c120d2a763">separate study</a> by Lawi, Panggabean, and Yoshida, found REST was faster on response time and throughput in that test setup, while GraphQL used about <strong>37-40%</strong> less CPU and memory. That supports a real resource-efficiency tradeoff, though the result is specific to that workload.</p><p>Elghazal, Aneiba, and Shahra presented at <a href="https://www.scitepress.org/Papers/2025/137299/137299.pdf">WEBIST 2025</a> and found REST ~3.7x faster on response time in a Go-based microservices setup, while GraphQL transferred significantly less data overall, reducing total data volume from several gigabytes to a few hundred megabytes. The 3.7x latency gap is larger than what other studies report, which suggests strong sensitivity to implementation and workload.</p><h4><strong>Across these studies, a general pattern emerges</strong></h4><p>Performance varies by workload, and neither approach is consistently faster. REST often has lower per-request overhead for simple payloads and at high concurrency. GraphQL can have lower total latency when it reduces or eliminates multiple round-trips, transfers less data, or uses fewer server resources.</p><p>The right choice depends on the workload, including schema design, query patterns, and implementation details. Citing a flat CRUD benchmark to evaluate a system designed for relational queries is technically correct, but it measures a different use case.</p><p>How are teams governing schema changes, handling production traffic, and measuring Federation success? Share your experience and get early access to the full report. For every valid survey completed, we'll donate $30 to <a href="https://www.unicef.org/">UNICEF</a>.</p><h2>GraphQL Caching vs REST Caching: The Real Tradeoffs</h2><h3>Claim 8: &quot;GraphQL uses POST to a single endpoint; HTTP caching does not work&quot;</h3><p><strong>Verdict: Partially true (default vs production setups).</strong></p><p>Standard GraphQL implementations use POST to <code>/graphql</code>. POST requests are not cached by browsers, CDNs, or reverse proxies by default. This is a real limitation and worth understanding.</p><p>HTTP caching relies on two things: the request URL being stable across clients, and the HTTP method being GET in most common caching setups. When every GraphQL request is a POST with a body that is not used as part of the cache key by intermediaries, caches cannot reliably match requests to stored responses.</p><p>More precisely, the issue is a lack of stable cache keys rather than POST itself, since POST responses can be cached when explicitly configured. By default, this gives REST an advantage for simple, per-resource caching, while GraphQL requires additional setup to achieve similar behavior.</p><p>But the comparison with REST is asymmetric.</p><p>As we established in Claim 5, REST serving relational data often either makes multiple individual GET requests (cacheable but N+1 round-trips) or uses a batch endpoint (efficient but less cacheable in practice). In many production systems, REST avoids this trade-off by relying on per-resource caching combined with client-side or edge aggregation (for example, a BFF or gateway), rather than batching at the CDN layer.</p><p>A REST batch endpoint like <code>GET /users/posts?ids=1,2,3,...,25</code> produces a different URL for every unique combination of requested users. And as we noted in Claim 5, it's not just the combination that matters, but also the order: <code>?ids=1,2,3</code> and <code>?ids=3,1,2</code> are different cache keys to a CDN. CDN cache hit rates for these parameterized URLs can be lower in practice, depending on usage patterns.</p><p>Even if you do manage to cache batch URLs, invalidation becomes expensive. When a single entity changes (say, user 7 updates their profile), you need to invalidate every cached batch URL that contains that user's ID. Most CDNs have no built-in way to compute which URLs contain which IDs without additional tagging or indexing. A single entity change can invalidate an unpredictable number of cached responses. This comparison mixes HTTP-level caching with client-side caching, which solve different problems but are often combined in real systems.</p><p>Compare this to GraphQL's normalized client cache (Apollo Client, Relay), where invalidation happens at the entity level. This operates at the client layer and does not replace HTTP-level caching. User 7's data is stored once by its cache ID. When it changes, only that entity updates, and every view referencing it refreshes automatically. No URL permutation problem. No batch invalidation cascade.</p><p>These trade-offs can make it difficult to simultaneously optimize REST for both batch efficiency (for the N+1 comparison) and high HTTP cache reuse (which requires stable, per-resource URLs). These approaches are in tension. Optimizing for batch efficiency reduces round-trips, but tends to lower CDN cache hit rates and complicate invalidation.</p><h4><strong>A common production solution: Persisted Queries.</strong></h4><p>Also called trusted documents, <a href="https://benjie.dev/graphql/trusted-documents">persisted queries</a> are a recommended production pattern for GraphQL APIs. Instead of sending the full query string in a POST body, the client sends a hash: <code>GET /graphql?extensions={&quot;persistedQuery&quot;:{&quot;sha256Hash&quot;:&quot;abc123&quot;}}</code>.</p><p>This approach is most effective for first-party APIs, where the set of allowed operations can be controlled.</p><p>It can be implemented as a GET request with a more stable URL when queries are pre-registered. This requires maintaining a registry of allowed operations and coordinating deployments between client and server. CDNs can cache it like other GET requests when configured appropriately.</p><p>However, this approach is less flexible for highly dynamic queries and is harder to apply in public or third-party API scenarios. Variables still affect cache keys, so different inputs produce different cache entries, and fragmentation can still occur depending on usage patterns. <a href="https://experienceleague.adobe.com/en/docs/experience-manager-cloud-service/content/headless/deployment/dispatcher-caching">Adobe Experience Manager</a>, for example, recommends persisted queries as the preferred way to enable CDN caching for GraphQL.</p><p>Persisted queries can help address the caching problem (GET requests, stable URLs), the security problem (only pre-registered queries execute), and the client size problem (works with plain <code>fetch()</code>).</p><p>While some systems allow GET with query‑string GraphQL, it is brittle in practice, which is why POST‑by‑default is the effective pattern.</p><h3>Claim 9: &quot;56% of teams report caching challenges with GraphQL&quot;</h3><p><strong>Verdict: Partially true (source unclear, underlying concern valid).</strong></p><p>The <strong>56%</strong> figure is widely cited, attributed variously to an &quot;Apollo 2024 survey&quot; or a &quot;JetBrains Developer Survey.&quot; I could not locate a clear primary source for this exact number.  Given the lack of a verifiable primary source, this statistic should be treated as anecdotal rather than evidence‑based.</p><p>Even if the number were accurate, it would still need to be interpreted carefully. Regardless of that specific percentage, the underlying concern is real: GraphQL caching is more complex by default, because it often handles multi-resource queries that do not map cleanly to per-resource HTTP caching and pushes more responsibility onto the client and gateway layers. This means teams must make explicit architectural choices to get effective caching, rather than relying on default HTTP behavior.</p><p>REST systems can avoid some of this complexity through per-resource caching, but face their own trade-offs when aggregating data or batching requests, including reduced cache reuse and more complex invalidation. At the same time, the problem is well understood, and there are established patterns to address it: persisted queries for CDN caching (see Claim 8), and normalized client caches for application-level caching (see Claim 10).</p><h3>Claim 10: &quot;Apollo Client weighs 43 KB gzipped&quot;</h3><p><strong>Verdict: Misleading comparison (incomplete framing).</strong></p><p>I often see comparisons framed as Apollo Client (43 KB) versus <code>fetch()</code> (0 KB), implying GraphQL requires a heavy client library while REST uses native browser APIs.</p><p>Historically, measurements around 40–45 KB gzipped were reasonable for older Apollo Client <a href="https://www.apollographql.com/docs/react/development-testing/reducing-bundle-size">bundles</a>, but recent v4.x releases have reduced bundle size and made features more modular. Regardless, that is not a cost of GraphQL itself. The same concern exists for feature‑rich REST clients like <a href="https://swr.vercel.app/">SWR</a> or <a href="https://tanstack.com/query/latest">TanStack Query</a>. It is a misleading comparison because both REST and GraphQL can be implemented with nothing more than <code>fetch()</code>. Therefore, the theoretical baseline for both styles is 0 KB.</p><p>GraphQL is an HTTP request:</p><pre>const data = await fetch('/graphql', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ query: '{ users { name } }' })
}).then(r =&gt; r.json())
</pre><p>No library is strictly required.</p><p>With persisted queries, it can be simpler at runtime: a GET request with a hash parameter. This requires pre-registering operations and coordinating between client and server.</p><p>In practice, many GraphQL applications adopt libraries like Apollo Client or Relay because they solve common problems around caching, state management, and request orchestration.  The additional bundle size in libraries like Apollo Client pays for these capabilities.</p><p>Similar trade-offs exist in REST clients, though GraphQL clients often include more built-in behavior due to the structure of the query model. In both cases, these libraries shift complexity from the API layer to the client. Without it, you often end up rebuilding similar logic yourself or pulling in REST-side alternatives like TanStack Query or SWR.</p><p><a href="https://www.apollographql.com/docs/react/">Apollo Client</a> provides a normalized cache that stores entities by ID and enables automatic UI updates when cached data changes. When a mutation returns updated data, Apollo can update the cache and reflect those changes in many relevant queries without requiring manual refetching.</p><p>On top of the normalized cache, Apollo provides additional features.<br />These include optimistic updates, pagination helpers, request deduplication, and partial error rendering.<br />It also includes local state management and DevTools for inspecting cache data and queries.</p><p><a href="https://relay.dev/">Relay</a> (by Meta) provides additional compile-time guarantees. Its ahead-of-time compiler runs at build time, catching malformed queries before the app ships. Fragment colocation lets each component declare the data it needs, and data masking ensures components only access the data defined in their fragments, preventing accidental coupling between components.</p><p>Relay includes configurable garbage collection for unreferenced records in its normalized store, helping limit memory growth in long-running applications. These features address common data consistency and state management problems.</p><p>In applications where the same entity appears in multiple places (e.g., avatar, comments, profile), a normalized cache with automatic mutation propagation reduces stale-data bugs. Building this yourself means maintaining a manual entity store with subscription and notification logic.</p><p>That's the role these libraries serve.</p><p>Real-world REST applications pay similar costs, though they can sometimes defer this complexity longer depending on access patterns. TanStack Query, Axios, and SWR introduce additional bundle size for features like caching and request management.</p><p>A fairer comparison is either naive-to-naive (no clients, just <code>fetch()</code>) or full-featured to full-featured: Apollo Client and tools like TanStack Query both add bundle size in exchange for caching, deduplication, and state management.</p><h3>Claim 11: &quot;fetch() ships with every browser at 0 KB&quot;</h3><p><strong>Verdict: Confirmed.</strong></p><p>True. The Fetch API is built into all modern browsers and is available natively in recent versions of Node.js. But as we showed in Claim 10, GraphQL also works with <code>fetch()</code> at 0 KB. This is often presented as a REST advantage, even though it applies equally to both.</p><h2>Is GraphQL Less Secure Than REST?</h2><h3>Claim 12: &quot;A 128-byte nested query can consume 10 seconds of CPU time&quot;</h3><p><strong>Verdict: Partially true (context-dependent).</strong></p><p>Security research shows that small, deeply nested GraphQL queries can consume disproportionate CPU time.</p><p>This applies primarily to resolver-based implementations without depth or cost controls. In a GraphQL schema with circular references (e.g., a <code>User</code> has <code>friends</code> who are also <code>User</code> objects), an attacker can construct a query like:</p><pre>{ user { friends { friends { friends { friends { friends { ... } } } } } } }
</pre><p>In a worst case scenario with an unoptimized implementation, if each user has 10 friends, 6 levels of nesting could theoretically result in up to 10^6 = 1,000,000 resolver calls. The query string itself is tiny, but the computational cost can grow quickly — in some cases exponentially — depending on schema design and resolver behavior.</p><p>The specific numbers depend on schema complexity and server hardware, but the principle is documented in <a href="https://medium.com/@ibm_ptc_security/denial-of-service-attacks-with-graphql-77189a6ba85b">security writeups and research</a>.</p><h4><strong>Scope: first-party vs public APIs matters.</strong></h4><p>For first-party APIs, trusted documents can significantly reduce this attack vector in controlled environments, but do not address inefficient resolver behavior within allowed operations. When a GraphQL server only executes pre-registered operations, arbitrary queries, including malicious depth attacks, can be rejected before execution. The <a href="https://graphql.org/learn/security/">official GraphQL.org Security page</a> confirms this.</p><p>For public APIs, this remains a real concern. GraphQL.org explicitly states that &quot;trusted documents can't be used for public APIs because the operations sent by third-party clients won't be known in advance.&quot; Public APIs need depth limiting, cost analysis, and rate limiting. Many implementations also use batching techniques (e.g., DataLoader) to reduce redundant resolver execution.</p><p>The attack scenario is valid for public APIs. But many GraphQL deployments are first-party, and persisted queries can prevent arbitrary depth attacks by restricting execution to pre-registered operations. These risks depend heavily on whether the API accepts arbitrary queries or restricts execution to known operations.</p><h3>Claim 13: &quot;80% of GraphQL APIs are vulnerable to denial-of-service through query depth&quot;</h3><p><strong>Verdict: Misleading framing (misinterpreted statistic).</strong></p><p>The <strong>80%</strong> figure is commonly misinterpreted from <a href="https://escape.tech/blog/the-state-of-graphql-security-2024/">Escape's State of GraphQL Security 2024</a>.</p><p>The reported DoS vulnerability rate is <strong>69%</strong> of scanned public GraphQL endpoints. <strong>80%</strong> comes from a separate finding in the same report: &quot;<strong>80%</strong> of issues could have been resolved by implementing best practices&quot;, which measures remediability, not prevalence.</p><p>The <strong>69%</strong> and <strong>80%</strong> figures come from different parts of the same report and measure different things; <strong>they are not interchangeable</strong>.</p><p>Public APIs accepting arbitrary third-party queries have a different security profile than first-party APIs using trusted documents. For first-party APIs, the depth-DoS attack vector can be significantly reduced when using persisted queries, though inefficient or overly expensive allowed operations can still create performance risks.</p><h4><strong>API security challenges are not unique to GraphQL.</strong></h4><p><a href="https://salt.security/blog/increasing-api-traffic-proliferating-attack-activity-and-lack-of-maturity-key-findings-from-salt-securitys-2024-state-of-api-security-report">Salt Security's 2024 report</a>, based on a survey of 250 IT and security professionals, found that <strong>37%</strong> experienced an API security incident in the past 12 months (up from <strong>17%</strong> in 2023), <strong>95%</strong> encountered security issues in production APIs, and <strong>23%</strong> suffered an API-related breach.</p><p>These findings apply to APIs broadly and are not specific to GraphQL.</p><p><strong>Important caveat:</strong> Escape's <strong>69%</strong> measures vulnerability scan results on 160 public endpoints. Salt's figures measure self-reported incidents across 250 organizations. They are not directly comparable because different methodologies are being used to measure different things. Even so, the broader point is: API security is an industry-wide challenge, not specific to GraphQL.</p><h3>Claim 14: &quot;Most frameworks ship with no default depth limit&quot;</h3><p><strong>Verdict: Partially true (implementation defaults).</strong></p><p>Many GraphQL server frameworks do not enable depth limiting by default. The <a href="https://graphql.org/learn/security/">official GraphQL.org Security page</a> recommends it, stating: &quot;Even when the N+1 problem has been remediated through batched requests to underlying data sources, overly nested fields may still place excessive load on server resources.&quot; This is not defined at the GraphQL specification level and is left to server implementations.</p><p><strong>This is a real concern and should be taken seriously, even with DataLoader or query compilation.</strong></p><p>The plugin ecosystem makes it relatively easy to add these controls. Libraries such as <a href="https://github.com/Escape-Technologies/graphql-armor">GraphQL Armor</a> and <a href="https://the-guild.dev/graphql/envelop">Envelop</a> plugins provide depth and cost controls. But it is opt-in, not opt-out.</p><p>However, the comparison should be symmetric.</p><p>Many REST frameworks also ship with limited security controls enabled by default. <a href="https://expressjs.com/">Express.js</a>, a minimal web framework, does not include rate limiting or input validation out of the box and relies on middleware for these concerns. <a href="https://www.django-rest-framework.org/">Django REST Framework</a> includes throttling features, but they are not enabled by default.</p><p>&quot;Insecure defaults&quot; is not a GraphQL-specific problem; it is a common pattern across web frameworks.</p><h2>HTTP Behavior and Error Handling</h2><h3>Claim 15: &quot;GraphQL returns HTTP 200 always, even when errors occur&quot;</h3><p><strong>Verdict: Misleading framing.</strong></p><p>This conflates GraphQL (the query language and execution engine) with one specific transport binding.</p><p>GraphQL is designed to be transport-independent. It runs over HTTP, WebSockets, Server-Sent Events, multipart responses, and other transport mechanisms. <strong>This is a deliberate architectural choice.</strong> GraphQL is similar to SQL in that it is a query language and does not dictate how queries are transported. You can run SQL over a TCP socket, a named pipe, or an HTTP API. GraphQL works the same way.</p><p>GraphQL reports errors in the response body's <code>errors</code> array rather than relying solely on transport-level status codes. A single GraphQL response can contain both <code>data</code> and <code>errors</code>, representing partial success.</p><p>The current working draft of the <a href="https://spec.graphql.org/draft/#sec-Handling-Field-Errors">GraphQL specification</a> defines this explicitly: a field that errors out returns <code>null</code> and adds an entry to the <code>errors</code> array, while sibling fields resolve successfully.</p><p>This is a defined behavior of the GraphQL execution model rather than a limitation. HTTP status codes do not cleanly express partial success: a request is typically represented as either 200 OK or 4xx/5xx, but not both.</p><p>GraphQL can tell a client &quot;here's <strong>90%</strong> of what you asked for, and here's exactly which field failed and why.&quot; In practice, many GraphQL-over-HTTP implementations return HTTP 200 responses with errors encoded in the response body. However, transport-level errors (e.g., invalid requests, authentication failures, or server errors) may still return appropriate non-200 status codes.</p><p>Under the commonly used <code>application/json</code> media type, the <a href="https://graphql.github.io/graphql-over-http/">GraphQL-over-HTTP specification</a> recommends returning 200 OK for any well‑formed GraphQL response, even when the <code>errors</code> field is populated. More nuanced status codes, including non-200 responses, are defined when using the <code>application/graphql-response+json</code> media type.</p><h3>Claim 16: &quot;The OWASP GraphQL Cheat Sheet reads like a confession of design decisions that were never made&quot;</h3><p><strong>Verdict: Misleading framing (not unique to GraphQL).</strong></p><p>The <a href="https://cheatsheetseries.owasp.org/cheatsheets/GraphQL_Cheat_Sheet.html">OWASP GraphQL Cheat Sheet</a> documents standard security hardening practices, similar to other OWASP cheat sheets. But OWASP publishes similar cheat sheets for many technologies.</p><p>The <a href="https://cheatsheetseries.owasp.org/cheatsheets/REST_Security_Cheat_Sheet.html">REST Security Cheat Sheet</a> covers input validation, output encoding, HTTPS enforcement, access control, rate limiting, and error handling. These are not defined by REST itself, but are implemented at the framework or application level. The <a href="https://cheatsheetseries.owasp.org/cheatsheets/REST_Assessment_Cheat_Sheet.html">REST Assessment Cheat Sheet</a> goes further, documenting how to test REST APIs for vulnerabilities.</p><p><a href="https://owasp.org/">OWASP</a> also publishes cheat sheets for <a href="https://cheatsheetseries.owasp.org/cheatsheets/Nodejs_Security_Cheat_Sheet.html">Node.js</a>, <a href="https://cheatsheetseries.owasp.org/cheatsheets/Django_Security_Cheat_Sheet.html">Django</a>, <a href="https://cheatsheetseries.owasp.org/cheatsheets/Ruby_on_Rails_Cheat_Sheet.html">Ruby on Rails</a>, and dozens of other technologies.</p><p>Characterizing the GraphQL cheat sheet as a &quot;confession&quot; while ignoring equivalent guidance for REST can imply that GraphQL is uniquely insecure compared to other technologies. The existence of an OWASP cheat sheet reflects community attention to security and ecosystem maturity, rather than indicating an inherent design flaw in the technology itself.</p><h2>Adoption and Tooling</h2><h3>Claim 17: &quot;83% of web services use REST&quot;</h3><p><strong>Verdict: Misleading framing (usage ≠ exclusivity).</strong></p><p>The <strong>83%</strong> figure is often attributed to the RapidAPI Developer Survey 2024, which other sources describe as measuring public APIs using REST, but I was not able to locate the original survey report directly. The <a href="https://www.postman.com/state-of-api/2025/">Postman 2025 State of the API Report</a> reports REST usage at <strong>93%</strong> among surveyed developers, not <strong>83%</strong>. Postman's survey allows multiple selections, so <strong>93%</strong> reflects developers who use REST, not developers who use REST exclusively.</p><p>The same survey shows <strong>33%</strong> of developers use GraphQL alongside REST, making it one of the most commonly used API styles in the survey. Multiple factors contribute to REST's widespread use beyond active evaluation:</p><ul><li><strong>Historical momentum:</strong> REST has been the default since the mid-2000s. Most APIs were built before GraphQL was available or widely adopted.</li><li><strong>GraphQL adoption is increasing:</strong> <a href="https://www.gartner.com/en/documents/7488253">Gartner has predicted</a> that by 2027, more than <strong>60%</strong> of enterprises will use GraphQL in production, up from less than <strong>30%</strong> in 2024.</li><li><strong>They coexist:</strong> Companies such as GitHub, Shopify, Netflix, Airbnb, Twitter, and The New York Times run GraphQL APIs in production, often alongside REST.</li></ul><p>Using REST's market share as evidence against GraphQL reflects historical adoption patterns more than a direct comparison of technical fit or capability.</p><h3>Claim 18: &quot;REST with OpenAPI 3.0 offers self-documenting, typed client generation, HTTP caching built in&quot;</h3><p><strong>Verdict: Misleading framing (omits comparable GraphQL capabilities).</strong></p><p>Everything stated about OpenAPI is true. However, GraphQL has comparable capabilities in each area, with different trade-offs.</p><p><strong>Self-documenting:</strong> A GraphQL server requires a GraphQL schema. The schema is not optional, not a nice-to-have, not a separate file you maintain on the side. It is a required part of the server.</p><p>GraphQL servers can expose their full type system through <a href="https://graphql.org/learn/introspection/">introspection</a>, which lets any client query available types, fields, and arguments at runtime. Meta's <a href="https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/">engineering blog</a> describes this as a core design goal: &quot;Introspective: A GraphQL server can be queried for the types it supports.&quot;</p><p>With REST, an OpenAPI spec is optional. You often need to generate or maintain the spec separately and keep it in sync with your implementation. Some REST APIs still ship without a machine-readable description. An OpenAPI spec can drift from the implementation because they are separate artifacts.</p><p>In GraphQL, the schema is part of the server itself, which in practice can make drift much harder, because the schema is tied directly to the running implementation, though that is not guaranteed.</p><h4><strong>Typed client generation</strong></h4><p><a href="https://the-guild.dev/graphql/codegen">GraphQL Codegen</a> (by The Guild) and tools like <a href="https://github.com/Khan/genqlient">genqlient</a> produce compile-time validated, fully typed client code. <a href="https://graphql.org/blog/2024-09-19-codegen/">GraphQL.org's blog</a> documents that operation-based codegen produces types specific to each query, which can be more precise than endpoint-level types because you get types matching exactly the fields you requested.</p><p>OpenAPI also supports typed client generation, but typically relies on maintaining a separate specification. In many cases, OpenAPI-based clients are simpler to generate and require less build-time integration.</p><h4><strong>HTTP caching</strong></h4><p>As discussed in Claims 8-9, persisted queries can enable HTTP caching for GraphQL. And as discussed in Claim 5, REST's HTTP caching can be less effective for some relational or batch-style queries, which is the use case being compared.</p><p>By default, GraphQL does not align as directly with HTTP caching as REST.</p><h3><strong>Totals (by verdict): 3 confirmed, 6 partially true, and 9 misleading or incomplete.</strong></h3><p>In practice, REST remains a strong default for simpler, resource‑oriented services, while GraphQL shines in complex, client‑driven, multi‑resource scenarios. The best choice depends on the workload, not on which protocol is “better” in the abstract.</p><h2>Part II: What the Evidence Means</h2><p>This section is opinionated. Part I was about what the sources say. This part is about what I think it means for the industry based on my experience building API infrastructure for the past several years. Take it as one perspective among many.</p><h2>Where REST Genuinely Wins</h2><p>REST is the right choice for a lot of problems.</p><ul><li>Simple CRUD with flat resources.</li><li>Public APIs where every resource has a stable URL.</li><li>Systems where HTTP caching on individual endpoints matters.</li><li>Scenarios where the data model is straightforward, and each client needs more or less the same response shape.</li></ul><p>REST's ecosystem is mature, and every developer knows that. The tooling is battle-tested. OpenAPI provides typed client generation and machine-readable documentation. For these use cases, adding GraphQL would be adding complexity without proportional benefit.</p><p>If you have 12 REST endpoints serving flat resources to a single frontend, and your team is productive, keep what you have.</p><p>If you have a single frontend talking to a single backend, and both are TypeScript, do you need REST at all? Tools like <a href="https://trpc.io/">tRPC</a> give you end-to-end type safety with zero schema definition, no OpenAPI spec, and no code generation step. Just TypeScript functions on the server, typed calls on the client. For that specific architecture, tRPC is simpler than both REST and GraphQL. The &quot;12 endpoints and a fetch() call&quot; scenario might not need REST any more than it needs GraphQL.</p><h2>Where GraphQL Genuinely Wins</h2><p>The problems GraphQL was designed to solve are real, and REST still has no good answer for them.</p><p><strong>Relational data across multiple sources.</strong> When a single client view needs data from users, orders, products, and reviews, GraphQL fetches it in one typed request. REST either makes multiple round-trips or requires purpose-built batch endpoints that are bespoke, uncacheable, and expensive to maintain.</p><p><strong>Self-describing schemas and codegen.</strong> The GraphQL schema is the documentation. Because it is tightly coupled to the implementation, that makes drift much harder in practice. <a href="https://the-guild.dev/graphql/codegen">GraphQL Codegen</a> produces compile-time validated, fully typed client code specific to each query. OpenAPI can do similar things, but requires maintaining a separate spec file.</p><p><strong>Mobile and bandwidth-constrained clients.</strong> This is the original problem Facebook solved in 2012. Clients request exactly the fields they need. No over-fetching and no under-fetching. The peer-reviewed study by (<a href="https://dl.acm.org/doi/10.1145/3357141.3357149">Seabra et al.</a>) found that GraphQL improved performance in two out of three tested applications under moderate workloads, but it did not outperform REST consistently, especially under higher load.</p><h2>GraphQL Federation Changes the Question Entirely</h2><p>You may see the argument framed as &quot;12 REST endpoints versus GraphQL complexity.&quot; That framing assumes your API surface stays small. For many organizations, it won't.</p><p><a href="https://graphql.org/learn/federation/">GraphQL Federation</a> allows organizations to start with a single monolithic GraphQL API and gradually decompose it into independently owned subgraphs as teams grow. All subgraphs compose into a unified supergraph that clients consume as one schema.</p><p><a href="https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/federation">Apollo Federation</a> pioneered this pattern, and today multiple vendors provide production-ready Federation routers: <a href="https://www.apollographql.com/docs/graphos/routing">Apollo Router</a> (Rust), <a href="https://cosmo-docs.wundergraph.com/">WunderGraph Cosmo Router</a> (Go, open-source Apache 2.0), <a href="https://the-guild.dev/graphql/hive">The Guild's Hive Gateway</a> (TypeScript), <a href="https://chillicream.com/">ChilliCream's Hot Chocolate</a> (C#/.NET), and <a href="https://aws.amazon.com/appsync/">AWS AppSync</a> (managed service). Competition between vendors means teams have choices without being locked into a single provider.</p><p>The <a href="https://github.com/graphql/composite-schemas-wg">Composite Schemas Working Group</a>, an official subcommittee of the GraphQL Foundation, is working to standardize how subgraph schemas compose into a unified API.</p><p>The <a href="https://graphql.github.io/composite-schemas-spec/">specification</a> defines vendor-neutral composition and distributed execution rules, so that subgraphs written against the spec work with any compliant router. This is Federation becoming an open standard, not a single vendor's product.</p><p>REST has no equivalent. When REST-based organizations need to compose multiple services behind a unified API, they choose between API Gateways (vendor-specific routing), BFF patterns (bespoke aggregation code per frontend), or OpenAPI merging (flat endpoint lists with no cross-service entity relationships). Each approach is proprietary, and none handles the question: &quot;A User from Service A has Orders from Service B.&quot;</p><p><strong>Federation answers that question by design.</strong></p><p>And here is what I think many people miss: <strong>you can have Federation as the composition layer and REST as the consumption layer.</strong> Build a federated supergraph to unify APIs across teams. Then expose a REST API on top for clients that want it. This is a pattern we see growing in enterprise deployments. Federation and REST are not opposing choices. They operate at different levels of the stack.</p><h2>AI Agents Need Structured, Self-Describing APIs</h2><p>The API consumer is changing. Increasingly, the client calling your API is an <a href="https://wundergraph.com/blog/graphql-api-layer-for-ai-agents">AI agent</a>, not a human developer reading docs.</p><p>Agents need to discover capabilities, understand types, and request specific fields, all within a limited context window. GraphQL's typed, introspectable schema was built for exactly this interaction pattern. An agent can query the schema to understand what's available, then construct a precise request for exactly the data it needs.</p><p>REST with OpenAPI can serve agents, too. But exposing thousands of REST endpoints to an agent overwhelms its context window. A structured graph lets agents search and select precisely what they need.</p><p>The industry is moving in this direction. The <a href="https://graphql.org/blog/2026-01-20-recap-nov-2025-ai-wg/">GraphQL AI Working Group</a> is developing standards for how LLM agents interact with GraphQL APIs.</p><p>That said, AI agents will also call REST APIs. Tools like MCP are protocol-agnostic.</p><p>The question is how you organize your APIs so agents can find what they need. A federated supergraph provides that organization. A flat list of REST endpoints does not.</p><h2>The Right Question Is Not Which One Wins</h2><p>The REST vs. GraphQL debate keeps producing articles that pick one side and build the case. Some pick REST. Plenty of others pick GraphQL. Both sides select favorable benchmarks, ignore inconvenient tradeoffs, and declare the other technology dead or dying.</p><p>The evidence does not support either extreme.</p><p>REST at <strong>93%</strong> adoption (<a href="https://www.postman.com/state-of-api/2025/">Postman 2025</a>) is not under threat from GraphQL. GraphQL, with <a href="https://www.gartner.com/en/documents/7488253">Gartner predicting 60%+ enterprise adoption by 2027</a> and <strong>33%</strong> of developers already using it alongside REST, is not a niche technology. They coexist because they solve different problems.</p><p>The right question is: what shape is your data, who are your consumers, and how will your API surface grow?</p><p>For public APIs where third-party developers need to integrate: REST. REST makes it easy to generate SDKs in any language, and every developer already knows how to call a REST endpoint. For flat resources, stable URLs, and simple CRUD: also REST. For a single TypeScript frontend and backend: maybe tRPC. For relational data, multiple consumers with different needs, and cross-team composition: GraphQL. For organizations scaling from one team to many: Federation. For all of the above at the same time: both.</p><p>The next time you read a blog post that makes specific technical claims about either technology (including one of my posts), check the sources. Read the original documentation. Look at the methodology behind the benchmarks. The evidence is out there. Use it.</p><hr /><hr /></article>
